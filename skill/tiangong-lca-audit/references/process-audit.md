# 过程数据集审核方法

过程审核由四个窗口审核和跨窗口一致性审核组成。先完成核心检查，再处理表达和规范性问题。

## 1. 过程信息

检查：

- 名称、路线、位置类型、一般性说明是否指向同一对象。
- 分类是否与产品、工艺和用途存在可直接验证的冲突；分类判断必须按 `taxonomy-guide.md` 实际检索项目分类库，并在证据中记录检索词、命中候选和未采用候选的原因。
- 参考年份、地理位置、技术描述是否足以理解代表性。
- 一般性说明是否明确过程边界和目标产品。
- 一般性说明和技术描述是否区分来源文献的项目背景、核算结论或参数与本数据集的实际建模内容。

不要仅因分类不够精确而驳回。需要候选分类时读取 `taxonomy-guide.md`；分类类发现应区分分类路径直接匹配和仅示例名称匹配，无法确认唯一候选时给出候选范围和确认条件。

## 2. 建模信息

检查：

- 数据集类型是否与技术描述和清单边界匹配；数据集类型必须严格判断，不能把字段值当作默认可信结论。
- LCI 方法字段是否与数据集类型或正文冲突。
- 数据切断、完整性、来源与处理说明是否足以支撑关键清单。
- 多产品、联产品、系统扩展或分配角色是否说明清楚。
- 已引用来源文献时，先承认引用存在，再核查文献数据的处理说明以及说明字段是否准确表征当前数据集；不得把“使用文献内容”本身判定为问题。

覆盖多个装置或工序不自动意味着错误；关键是数据集类型和说明能否准确表达其黑箱或汇总边界。

### 数据集类型判定

按 CFIA-LCA 数据库导则的数据集组成口径判定：

| 类型 | 可接受证据 | 常见问题 |
| --- | --- | --- |
| 单元过程，单一操作 | 技术描述和清单只覆盖一个明确操作 | 覆盖多个装置、工序或厂内辅助过程却填单一操作 |
| 单元过程，黑箱 | 多个前景操作被聚合为一个过程，输入输出代表该黑箱边界 | 误填成单一操作，或未说明黑箱边界 |
| 部分终止系统 | 明确哪些上游或背景过程已终止、哪些输入仍调用背景过程 | 只填 Partly terminated system，但说明仍像普通制造过程 |
| LCI 结果/聚合数据集 | 清单表现为聚合生命周期结果，通常只保留参考产品与基本流 | 普通前景制造过程误填为聚合结果 |

如果数据集描述的是多个厂内工序共同生产一个产品，但没有说明上游过程已终止，优先判断为“单元过程，黑箱”。只有看见终止范围、剩余背景输入或聚合结果证据时，才接受“部分终止系统”或 “LCI 结果”。

### 边界、截断和回收口径

必须逐项核查：

- 生命周期边界词是否准确。若写“从摇篮到大门 / cradle-to-gate”，又写“不含运输、收集、预处理”等，应要求改成实际包含的过程范围，并列明排除活动。
- 截断原则不能只写系统边界。优先要求说明定量截断标准、覆盖率、排除活动和跨系统边界的流。
- 输入为废料、再生料、回收料或二次原料时，必须说明采用 cut-off、recycled-content、avoided burden 或其他口径，并说明收集、分选、预处理、运输和上游原生材料负荷是否纳入。
- 归因、归果、分配、系统扩展等方法字段不能替代数据处理说明。若涉及共用流或多产品，必须说明专属流、共用流、分配依据和目标产品参考量。

产品流与基准流是不同概念。过程可以输出多个产品流，但仅将其中一个指定为基准流；唯一基准流填写 100% 分配属于正常设置。不得仅因其他产品流不是基准流、未填写分配比例或名称包含“废料”而提出退回意见。只有存在多个实际承担过程负荷的联产品、已填写的分配字段彼此冲突，或数据集对分配方法作出与清单不一致的明确说明时，才核查并提出分配问题。

## 3. 管理信息

检查：

- 版本、格式、委托方或拥有者、录入人、永久标识是否可识别。
- 信息是否足以追溯和复用数据集。

工作流状态、注册号、访问限制等字段为空时，不单独判定不通过。

## 4. 输入/输出

检查：

- 核心输出是否存在，参考流、数量和单位是否明确。
- 主要原料、能源、产品、废物和典型排放是否与描述相符。
- 交换量和单位是否存在可验证的数量级、缩放或物理合理性问题。
- 多个有价值输出是否说明建模角色。
- 流名称、流类型、分类、方向和参考单位是否相互匹配。
- 外购、采购或作为原辅料消耗的输入应优先是产品流；若被建模为基本流或资源流，必须有证据说明其确为来自空气、水体或地质环境的环境交换。
- 流名称表达的形态是否与参考单位或参考属性匹配，例如液化/气态、溶液浓度、干基/湿基、质量/体积口径。
- 有大量水输入或水体排放时，是否说明循环水、蒸发损失、废水处理边界、污泥含水和外部处理关系。
- 内部循环、公用工程回收和外供产品是否被正确区分。

流显示“未审核”本身不是问题。已审核流若内容不合适，建议重新选择；未审核流可建议直接修改。
平台动态生成的 LCIA 结果默认不属于过程数据集退回审核范围，具体边界见 `audit-policy.md`。

## 5. 数据质量和文档合规

检查：

- 技术、地理、时间代表性是否与来源、参考年份、地理位置和技术描述一致。
- 是否有精度、完整性、一致性或 DQR 信息；缺失时不机械阻断，但若无法判断代表性或复用范围，应列为人工确认。
- 是否有目标用途、限制、受众、比较使用说明、委托方或拥有者。
- 是否能追溯数据来源、背景报告、审查报告、数据生成者、建模者或文档编制者；已有等效来源和拥有者时不要重复驳回。
- source 原文是否支持当前数据集的边界、年份、地区、技术路线、关键输入输出、截断和数据处理说明；若 source 只是项目背景、参数来源或 LCIA 结果，必须区分其与当前数据集自身建模内容。

source 核验由 Agent 语义判断完成，不由程序字符串匹配完成。执行步骤：

1. 先确认已取得可读的 PDF、全文、附录、source table、raw import 表或工程资料。对 PDF、Office、图片或复杂表格材料，必须直接调用项目内 `skill/document-granular-decompose` 生成 image-aware 全文；仅有 `pypdf` 抽取文本时，复杂表格、图片和补充材料相关事实不得视为已完整抽取。必须检查 `sources/*/manifest.json` 的 `related_artifact_requirements`，并主动检索主文中的 Supplementary Table、appendix、supporting information、source table 等引用；若主文指向 Supplementary Table S8 这类补充表但文件未取得，应继续从平台 source dataset、出版商/DOI 页面或文中 URL 获取。无法取得时记录缺失的具体字段，不得把主文未包含的数值判为已支持。
2. 读取 `source-checks/claims.json` 中待核验字段；claims 必须覆盖所有输入/输出交换，而不只是参考流。没有 claims 或发现 claims 漏掉交换时，从数据集名称、路线、位置、年份、数据集类型、技术描述、参考流和全部关键清单自行补充待核验字段。
3. 对每个字段同时读取数据集字段值、字段所在窗口说明、source 摘录或 PDF 原文上下文；必要时在 `sources/*/extracted.md` 中继续检索同义词、参数、表格标题和上下文。
4. 将字段拆成可核查事实，例如对象、数量、单位口径、基准流或 qref、流身份、地区、年份、技术路线、边界、分配、截断和数据处理口径；不要要求 source 出现平台字段的完整显示字符串。
5. 对表格或数值证据，记录原文表名、页码或 appendix、原始字段、原始单位、原始值、目标字段、换算规则和换算后值；例如 `t -> kg` 不能只在聊天中解释。
6. 逐项判断 source 是否支持这些事实，并记录 source ID、页码或表名、摘录、判断理由和状态。

source 核验结论分为：

- `matched`：source 原文内容能直接支持当前字段事实；允许中英文翻译、同义表达、表格表达或顺序不同。
- `conflict`：source 与当前字段存在直接冲突，例如数值、单位、对象、地区、年份、边界或数据处理口径不一致。
- `not_found`：source 已抽取并已检索相关上下文，但没有找到当前字段事实的支持证据。
- `ambiguous`：source 有相关信息，但不足以确认是否支持当前字段，例如只支持项目背景、不支持当前建模口径，或缺少补充表/处理说明。
- `source_unavailable` 或 `extraction_failed`：无法完成核验，只能限制结论或列为信息缺口。

不得把整段字段值的字符串命中、关键词命中或程序生成的候选结果当作最终 source 结论。程序只能准备待核验字段和 source 文本；最终 `matched`、`conflict`、`ambiguous`、`not_found` 必须来自 Agent 对文章和数据集的语义阅读。

## 6. 跨窗口一致性矩阵

| 对照关系 | 重点问题 |
| --- | --- |
| 名称 / 一般性说明 / 核心输出 | 是否审核同一个产品或服务对象 |
| 技术描述 / 数据集类型 | 单一操作、黑箱、汇总过程等角色是否一致 |
| 一般性说明 / 输入输出 | 明确包含或排除的流是否与清单冲突 |
| 边界表述 / 排除活动 | cradle-to-gate、gate-to-gate、厂内边界是否与不含运输、收集或预处理一致 |
| 废料输入 / 回收说明 | 二次原料负担和前处理边界是否可判断 |
| 功能单位或基准流 / 核心输出 | 参考流、结果量、单位和量级是否一致 |
| 多产品输出 / 分配说明 | 联产品角色和处理方法是否说明 |
| 数据来源 / 年份 / 地理 / 技术 | 代表性描述是否互相支撑 |
| 流名称 / 类型 / 分类 / 方向 | 是否存在可直接验证的语义矛盾 |
| 流名称 / 单位 / 形态 | 液态、气态、溶液、干湿基、质量和体积口径是否一致 |
| 水输入 / 水体排放 / 废水处理 | 水量去向和内外部处理边界是否清楚 |
| 中英文内容 | 对象、边界、数量和用途是否一致 |

## 7. 审核完成条件

只有完成对象、数据集类型、边界、功能单位或基准流、输入/输出、来源代表性和数据质量/文档可追溯性检查后，才可以给出完整通过结论。

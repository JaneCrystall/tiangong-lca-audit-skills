# 输出契约

审核输出应让审核员能够复核，也让提交者知道如何修改。

## 1. 标准结构

```text
## 证据型报告

审核范围：
数据集类型：
输入充分性：
第一层 PDF/source 一致性：
第二层规则符合性：
审核结论：

阻断问题：
1. [位置] 问题与证据。影响：……。建议：……。

建议修改：
1. [位置] 问题与证据。建议：……。

需人工确认：
1. [位置] 待确认事项、原因和需要补充的证据。

信息缺口：
1. 缺失内容及其阻止完成的审核项。

结论说明：

## 平台退回意见
```

没有内容的分组可以省略。

审核结论必须先分为两个判断层，再给综合结论：

- 第一层 `PDF/source 一致性`：回答数据集字段、年份、名称、地点、技术路线、参考流等 claims 是否能被 PDF/source 原文语义支持，或是否存在直接冲突。该层必须由 Agent 阅读字段和 source 上下文后判断；不得用整段字段值字符串命中、关键词命中或程序规则作为最终结论。结论可为 `一致`、`未完全证实`、`不一致`、`证据不足`、`需人工确认`、`未核验`。
- 第二层 `规则符合性`：回答数据集是否符合机器规则和审核政策，包括阻断问题、建议修改、人工确认项和信息缺口。结论可为 `符合规则`、`基本符合规则，有建议修改`、`不符合规则`、`规则证据不足`、`需人工确认`。
- 综合结论根据两层结论和 findings 聚合为 `通过`、`不通过`、`信息不足` 或 `需人工确认`。source 字段 `not_found` 不能自动等同于数据错误；只有直接冲突或规则阻断才进入不通过判断。

输出视图必须分为两类：

- `## 证据型报告` 面向审核员复核，可以保留规则编号、证据边界、信息缺口和人工确认项。
- `## 平台退回意见` 面向提交者修改，只写可定位、可执行的意见；证据不足或仅用于限制结论的信息缺口不得写成绝对结论。

source 核验结果在证据型报告中应写明 source ID、文件、页码或表名、核验字段、状态、摘录和一句判断理由。涉及表格或数值时，还应写明原始单位、原始值、目标字段和换算链；缺少附录/source table 时，应列出缺失字段而不是只写“需要 source”。只有 source 与数据集字段存在直接冲突，或 source 明确不能支持当前数据处理口径时，才写成平台退回意见。source 不可用、抽取失败或字段未命中通常限制结论或形成信息缺口，不得自动写成“数据错误”。

## 2. 批量审核与案件存档

- 批量任务必须遵循“一条数据集一份独立 Markdown 审核报告”；每份报告只包含该数据集的证据、发现、结论和平台意见。
- 可以额外生成批次汇总索引，但汇总索引不能代替独立报告，也不能把多条数据集的平台意见合并为一份报告。
- 每条数据集应独立存档完成审核所使用的原始输入或平台响应，包括审核任务、数据集详情和实际引用的关联流或关联过程。
- 存档用于后续复核、人工纠偏和回归评测；真实平台数据遵守仓库隐私要求，不进入 Skill，也不得提交。
- 审核报告和存档在未经人工确认或纠偏前，不得直接作为训练真值，也不得据此自动升级通用规则。
- 无法取得数据集详情时，仍为该任务生成独立报告并存档现有任务证据，结论按输入充分性规则处理。

## 3. 写作要求

- 先写事实，再写判断。
- 一条发现只表达一个可执行问题；同一根因不要拆成多条。
- 修改建议尽量控制为一句话，并明确修改字段或核查动作。
- 对学习者解释关键逻辑，但不写长篇 LCA 教程。
- 不使用“请规范填写”“请检查”等无法执行的空泛表述。
- 不使用“明显”“大量”“严重”等无判定标准的程度词；直接陈述冲突事实，能够量化时给出差值、比例或适用阈值。
- 不输出内部推理、置信度或为了凑数生成的问题。

## 4. 通过说明

通过时仍需概括：

- 数据集对象与边界。
- 功能单位、基准流或模型目标。
- 清单或模型结构完整性。
- 来源与代表性。
- 单位、分类和管理信息的总体情况。

通过说明不是正式签字。正式通过报告草稿使用 `assets/approval-report-template.md`。

## 5. 平台意见

每份审核报告末尾必须包含 `## 平台退回意见`，用于提供可直接复制到天工平台的审核文字；该部分只能来自前文证据型报告中的发现。

平台退回意见的写法：

- 使用 `①②③……` 连续编号，每条意见独立成段。
- 每条按“位置 + 可见证据 + 问题判断 + 明确修改建议”组织为一段完整文字。
- 直接使用页面中的字段名、流名称、分类和数值，使提交者能够定位问题。
- 当问题来自底层清单但可能未在页面正常展示时，明确写出交换方向、数量和流名称；不得用未经说明的工艺角色推断替代可见证据。
- 当问题来自 source 核验时，明确写出 source 名称或 ID、页码或摘录位置，以及它与当前字段的冲突；不得只写“与来源不一致”。
- 存在同名数据集时，争议交换必须同时核对数据集 ID、版本、交换方向、数量和流名称；报告中应写明足以区分记录的信息，不能仅凭数据集名称或近似数值判断。
- 不使用“阻断问题”“建议修改”“需人工确认”“规则编号”等内部审核标签。
- 不使用 Markdown 加粗、子项目符号、表格或需要二次整理的字段式结构。
- 不写空泛的“请检查”“请规范填写”；必须说明需要核实或修改的具体内容。
- 流分类或已审核流需要修改时，应写明实际检索到的候选分类路径或候选范围；无法确认唯一候选时，写明候选范围、需核实条件，以及“找到合适分类或者采用‘新增版本号’的方式对流分类进行修改”。
- 可将有明确修改价值的阻断问题、建议修改和人工确认项合并进平台意见；不机械复制仅由页面展示限制造成的信息缺口。
- 平台意见必须忠实于前文审核发现，不新增没有证据支撑的问题。

示例：

```text
## 平台退回意见

①输入/输出 中，流名称为“煤质活性炭; 活性炭，煤，高温蒸汽激活; 在工厂”，分类末级为“木炭 > 木炭”。流名称和分类冲突；分类库检索到的候选范围为“化学品 > 其他化学产品”或“木炭/活性炭相关产品”，需按平台实际分类树核实。建议找到合适分类或者采用“新增版本号”的方式对流分类进行修改。

②过程信息 / 技术描述及背景系统 中，英文为“pre-heating”，中文为“预整”，建议核实实际工序；若为预热，将中文“预整”改为“预热”，并保持中英文一致。
```

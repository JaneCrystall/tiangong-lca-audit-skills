# 审核判断政策

本文件定义所有过程与模型审核共同遵守的证据、严重程度和结论规则。

## 1. 证据优先级

1. **直接证据**：平台字段、输入/输出记录、模型结构、关联过程内容。
2. **跨字段证据**：多个窗口或多个关联过程之间可以直接验证的对应关系。
3. **规则依据**：本 Skill 的 `rules/*.json` 与审核方法。
4. **参考判断**：分类候选、行业常识和历史案例，只能帮助定位问题，不能单独证明问题。

每条审核发现至少需要一项直接证据。只有参考判断时，输出“需人工确认”。

规则中的 `standard_refs` 记录其对应的 CFIA-LCA-database-guideline 文件和章节，只作为规则依据锚点；审核输出不必逐条引用标准章节。若标准依据和本 Skill 规则发生冲突，先停止自动通过，按纠偏流程修正规则。

## 2. 发现类型

| 类型 | 含义 | 是否影响通过 |
| --- | --- | --- |
| `blocking` | 已有充分证据证明存在关键缺失、错误或矛盾 | 是 |
| `advisory` | 不影响当前通过，但能提升清晰度、规范性或复用质量 | 否 |
| `manual_review` | 可能影响结论，但规则、分类范围或业务事实不足以确认 | 暂不自动影响 |
| `input_gap` | 缺少完成核心检查所需的数据 | 可能导致信息不足 |

发现类型可根据实际证据调整，不得仅沿用规则默认值。

## 3. 结论聚合

按以下顺序生成唯一结论：

1. 存在一条或以上已验证的 `blocking`：**不通过**。
2. 不存在阻断问题，但核心审核维度因 `input_gap` 无法完成：**信息不足**。
3. 不存在阻断问题和核心信息缺口，但存在可能改变结论的 `manual_review`：**需人工确认**。
4. 核心审核维度已完成且不存在阻断问题：**通过**。

只有 `advisory` 时仍可通过。

## 4. 必填与非必填

- 必填字段先检查是否存在，再检查内容是否正确。
- 非必填字段为空通常不产生发现。
- 非必填字段一旦填写，若与核心字段或输入/输出冲突，可以形成阻断问题。
- 某字段是否必填不明确时，不机械驳回；记录为人工确认。

## 5. 防止过严

- 分类没有完全契合项时，合理近似可以接受。
- 单元过程、单一操作或黑箱过程的 LCI 方法字段为空、`-` 或“不适用”通常可以接受；只有与数据集类型或正文冲突时才提出问题。
- `Calculated` 不等于不可追溯；已有来源或处理报告时，只核查关键计算是否需要补充。
- 已提供来源文献时，不得表述为“未标注来源”；继续核查时应准确指出缺失的是引用位置、数据处理说明，还是文献内容与数据集实际建模内容的区分。
- 来源文献的项目描述、参数和核算结论可以作为数据来源，但不得在未加区分时被当作当前数据集自身的描述、参数或结果。
- 平台动态生成的 LCIA 结果可能随背景过程链接、审核状态或平台计算更新而变化；默认不将其作为退回依据，也不要求提交者统一或重新计算该结果。只有审核任务明确要求核查计算结果且计算上下文稳定、完整时，才对其形成单独判断。
- 流的“未审核”状态不等于流内容错误。
- 产品流不等于基准流；非基准产品流未填写分配比例本身不构成问题，唯一基准流填写 100% 分配属于正常设置。
- 只看到部分窗口时，不得形成完整通过结论。
- 平台状态为“已通过”或存在历史通过记录，不等于本轮语义审核通过；仍需完成对象、类型、边界、功能单位或目标量、清单或模型结构、来源代表性和数据质量检查。
- 数据集类型是核心方法字段。`Partly terminated system`、`LCI result` 等类型必须有相应边界证据；没有证据时不得按字段值默认通过。

## 6. 发现记录结构

每条发现包含：

- `location`：窗口、字段、流或关联过程。
- `evidence`：输入中可复核的事实。
- `judgment`：为何构成问题或为何需确认。
- `severity`：四类发现之一。
- `impact`：是否影响通过及影响原因。
- `suggestion`：一句话可执行修改建议。
- `rule_id`：适用规则；没有明确规则时可为空。
